Direct prompt injection
- paper: Jailbroken: How does LLM safety training fails - https://arxiv.org/abs/2307.02483
- coming from user input
- vulnerable because: competing objectives - LLM trained on competing objectives - prefix injection, role play, ...
                                           - example: tell me how ...  start with: "Sure, here is how" -
                      mismatched generalization - LLM trained on large, diverse dataset than safety training - encoding, locales, etc
                                                - example: pass encoded request like  V7hhdsad84fdasf4a68... (base64)
- can be improved with RLFH - Reinforcment-Learining with Human Feedback - human ranks by preferences during training
- attack - Jailbreak templates - exhausts context space and enforce 'own system prompt'
         - Crescendo - multi-turn attack - starts with harmless prompts and gradually steer to harmful output


Indirect prompt injection
- coming from data - external inputs
- for single turn attacks
- can affect not only output but also internal tools - like changing LLM persona
- example: email with prompt (can be hidden via white color font, base64 encoded), bank transfer with message, etc.
           shitposting on reddit and using for training - add glue to pizza (not attack by intention)

            - poisoned search result with instruction

Mitigation/Guardrails
- layered protection system
   - visibility layer - monitoring real time events
   - detection layer - automated tools to identify sus activity/anomalies
   - protection layer - input filtering, output validation, sandboxing
   - response layer - incident response plans, forensic analysis, system recovery

- Spotlighting
   - paper - Defending Against Indirect Prompt Injection Attacks With Spotlighting - https://arxiv.org/abs/2403.14720
   - family of prompt techniques to help distinguish between input sources (data vs user input)
   - easy implementation
   - techniques in spotlighting - delimiting
                                   - wraps user input with special characters, e.g. << this is user input >>
                                - data marking
                                   - adds special character after every input word, e.g. hello^word^
                                - encoding
                                   - encode user input in base64
   - sucks when system prompt is leaked (obvious how to bypass), but easy to implement
